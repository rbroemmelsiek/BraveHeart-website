# 06. Brave Heart Phase 2 CMS-Lite

🔴 **Complex** — Should be implemented only after Phase 1 site is stable.

## Objective

Add a lightweight WordPress-style content publishing system using markdown/frontmatter.

## Repo Inspection Preconditions (2026-06-21)

- Phase 1 routes, content, assets, CTAs, and SEO **must pass validation** (plans 01–05, 07) before starting.
- No blog infrastructure exists in `src/` today.
- `docs/braveheart/phase-2-cms-blog-workflow.md` defines intended workflow.
- **Open decision** in `PROGRESS.md`: content in file-based markdown only vs future database — default to **`src/content/braveheart/blog/`** or `content/blog/` with build-time import (Vite `import.meta.glob`).
- Sitemap generation from plan 05 must be extended, not replaced.

## Scope

- Blog index at `/blog`.
- Blog post route at `/blog/:slug`.
- Markdown/frontmatter parsing at build time.
- Categories and tags.
- Related links.
- Share metadata per post.
- Sitemap inclusion for published posts only.
- Publication status filtering (`draft` vs `published`).

## Out of Scope

- Full WordPress replacement.
- Database CMS.
- User roles or auth.
- Admin dashboard or WYSIWYG editor.
- Runtime AI title/tag generation (Cursor process only, per original plan).
- Modifying Phase 1 page content architecture.

## Files Likely Affected

- **New:** `src/content/braveheart/blog/*.md` (or agreed content folder)
- **New:** `src/pages/BlogIndex.tsx`, `BlogPost.tsx`, optional `TopicArchive.tsx`
- **New:** `src/lib/blog.ts` (or similar — frontmatter parser, slug index)
- `src/App.tsx` (Phase 2 routes)
- `src/components/Navbar.tsx`, `Footer.tsx` (blog link if approved)
- `public/sitemap.xml` or sitemap generator script
- `docs/braveheart/phase-2-cms-blog-workflow.md`
- `.agent/validation/braveheart-site-validation.md`
- `PROGRESS.md`

## Tasks

1. Confirm content folder location with owner; create structure and frontmatter schema.
2. Define frontmatter fields: title, slug, date, status, categories, tags, excerpt, ogImage, relatedLinks.
3. Implement build-time markdown loader (`import.meta.glob` + gray-matter or equivalent).
4. Implement blog index with published-post filtering.
5. Implement post route with slug validation and 404 fallback.
6. Implement category/tag archive routes if feasible (`/topics/[slug]` per route map).
7. Add SEOHead metadata per post from frontmatter.
8. Extend sitemap generation for published posts only.
9. Document Cursor-side auto-title/tag suggestion workflow (not runtime).
10. Add sample draft + published posts for validation.

## Validation

- Draft posts do not appear in blog index or sitemap.
- Published posts appear in blog index with correct titles/excerpts.
- Slugs are unique; duplicate slug build fails or warns.
- Category and tag archive pages render for test data.
- Related links resolve to valid internal routes or posts.
- OG metadata renders per post.
- Sitemap includes published posts only (not drafts).
- Phase 1 routes unaffected — regression spot-check all 10 routes.
- `npm run build` passes.
- `npm run lint` passes.

## Rollback Notes

Remove blog routes from `App.tsx`. Delete content folder and blog pages. Restore plan 05 sitemap to Phase 1 routes only.

## Risks Before Execution

- **Not single-pass feasible** — schema, loader, index, post, archives, and sitemap extension are multiple subsystems.
- Build-time glob may increase bundle size — consider lazy loading or separate chunks.
- Frontmatter parser adds dependency — evaluate `gray-matter` vs minimal custom parser.
- Category/tag URL design must match `route-map.md` before implementation.

## Approval Gate

Do not begin until Phase 1 validation report passes and owner explicitly approves Phase 2 start.
